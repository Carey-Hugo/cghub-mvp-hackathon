## 〇.1 🆕 提前量原则（死线前 24h 封板 + 48h 备份）

**核心铁律**：
1. **任何里程碑的"实际交付时间"必须早于"官死线"至少 24 小时**——给修复和补救留窗口
2. **关键节点同时设两个时间**：T = 官死线，T-24h = 内部封板线，T-48h = 备份完成线
3. **封板后变更必须走 Hugo 决策**，不再"改一改再说"

**关键节点提前量表**：

## 〇.2 🆕 补位机制（完不成时谁+怎么补）

**核心原则**：
- "完不成"不是指责，是信号
- 看到完不成的迹象 → **24 小时内** Hermes 主动介入
- 真正完不成 → **Hermes 和 Hugo 一起补位**（不是单纯催办）

**4 级响应**：

### 🟢 L1 · 正常推进（默认）
- 火堆按计划推进，群里每日 22:00 进展同步
- Hermes 只看不动手

### 🟡 L2 · 有风险（提前 48h 发现苗头）
**触发**：某火堆在 T-48h（备份完成线）还没动手
**动作**：
- Hermes 在群里公开点出"X 火堆在 T-48h 没动，**需要支援吗**"
- 主动 @ 火堆掌火人
- 提供具体可接手的工作（写代码、整文档、配环境）
- 让掌火人决定：**"我继续" or "Hermes 上手补"**

**触发词**（群里说）：`@掌火人 T-48h 备份线，X 没动，要不要我主动补？`

### 🟠 L3 · 落后（提前 24h 还在卡）
**触发**：T-24h（封板线）时某火堆没完成
**动作**：
- Hermes **直接动手补**，不等待
- 同步给 Hugo 私聊"X 落后，我主动补，预计 6h 内完成"
- 群内公告"X 任务由 Hermes 接手，原负责人转去 Y"
- Hugo 同步看，是否需要 Hugo 亲自补（战略/产品/演示方向）

**触发词**（群里说）：`@全体 X 任务超时，我主动补，转交原负责人去 Y`

### 🔴 L4 · 阻塞（差最后 12h 还是没成）
**触发**：T-12h 仍未完成，且影响整体提交
**动作**：
- Hugo + Hermes **同时上手**
- 群内拉**临时篝火**——找会做这个的人拉来群里
- 必要时**降低非核心需求**（砍 demo 演示场景、减少录屏时长、改 mock 数据）
- 8 人群里讨论：**必须放弃什么，保什么**（Hugo 决策）

**触发词**（群里说）：`@全体 L4 阻塞，Hugo + Hermes 同步上手，同时拉临时篝火`

### 补位实操清单（Hugo + Hermes 各自的补位方向）

| 任务 | Hugo 补位方向 | Hermes 补位方向 |
|------|--------------|----------------|
| 合约代码 | 战略决策（功能取舍）| 直接写 Solidity/Foundry |
| 前端代码 | 演示文稿 | 直接写 Next.js + wagmi |

| 截图证据 | — | 跑脚本 + 整理 |
| 提交 | **最终决策 + 提交** | 准备提交材料包 |
| 沟通协调 | **战略问题拍板** | 群内催办 + DM + 跨组协调 |

**原则**：Hugo 不写实现代码（除非非常紧急），**只做决策+关键产出**；Hermes 既能写代码也能做协调——**根据当时情况灵活切换**。

### 提前量 + 补位的触发器（Hugo 设给 Hermes 的红线）

Hermes 在以下时间点必须主动动作（不是等 Hugo 提醒）：

- **每日 22:00 进展同步**：检查每个火堆 T-48h 状态，L1→L2 升级
- **每日 12:00 节点检查**：检查 4 个火堆 T-24h 状态，L2→L3 升级
- **每周日 22:00 周复盘**：做整体风险评估，给 Hugo 提建议

如果 Hermes 自己判断"该升级了但还没升级"，**那是 Hermes 的失职**——Hugo 不用催，Hermes 必须自动。
