### R2 · Cobo Agentic Wallet App 配对未解决

**风险描述**：
- 华为 nova 5 + HarmonyOS 3.0 + 出境易 虚拟机 → Cobo Agentic Wallet App 闪退
- 配对不成功意味着 Pact auto-approved，**失去 Owner 审批环节**
- 评委可能质疑"Agent 是不是真的被人类控制"

**触发条件**：
- 6/10 前仍无法完成 App 配对
- 团队无海外 Android/iPhone 真机可用

**应对方案**（已分级）：
- **方案 A**（最佳）：找海外真机配对（团队成员或朋友）
- **方案 B**（已验证）：CLI auto-approved 模式跑通，Demo 文案强调"如需人工审批可加 Owner Pair"
- **方案 C**（兜底）：录屏时在脚本里说明"演示阶段使用 auto-approved，生产环境会启用 Pair"
- **方案 D**（演示技巧）：用脚本模拟 Owner 审批延迟，掩盖"无人工"问题

**关键证据**（已就位）：
- Sepolia TX `0x28015a...` 证明 CAW 跑通
- Policy Deny HTTP 403 证明服务端策略有效
- CLI 输出明确说明"未配对时 auto-approved"

**Owner**：Hugo
**状态**：✅ 缓解方案已落地
**监控频率**：每周一次

### R3 · Monad Testnet 部署不可用

**风险描述**：
- Monad Testnet 是新链，可能有 RPC 不稳 / 区块重组 / Faucet 限制
- AgentVault 参考项目跑通不代表我们能跑通
- 合约编译 / 部署 / 验证可能踩坑

**触发条件**：
- 6/6 Foundry 在 Monad 部署失败
- RPC 返回 503 / 超时

**应对方案**：
- **方案 A**（首选）：坚持 Monad，团队协同排查
- **方案 B**（已验证）：**回退 Sepolia** — CAW 已验证可用
- **方案 C**（备选）：用 Base Sepolia / Arbitrum Sepolia 等活跃测试网

**缓冲**：
- CAW 支持 9 条 EVM 链（ETH/SETH/BASE_ETH/MATIC/ARBITRUM_ETH/OPT_ETH/BSC_BNB/AVAXC/HYPEREVM_HYPE）
- 任意一条 EVM 链都能跑，灵活切换

**Owner**：白织 / Hugo
**监控频率**：每 2 天一次

---

### R4 · 端到端串联失败（Demo Day 前一天）

**风险描述**：
- 各模块单独能跑，串联起来失败（最常见的悲剧）
- 找不到 bug 在哪个边界
- 6/12 晚上还没跑通 → 6/13 中午要提交

**触发条件**：
- 6/12 18:00 端到端仍报错
- 错误信息涉及多个模块交叉

**应对方案**：
- **方案 A**（提前做）：**Day 7（6/10）就做第一次端到端串联**，留 3 天缓冲
- **方案 B**（分块隔离）：每加一个模块跑一次 e2e
- **方案 C**（最后兜底）：**只录屏能跑的部分**，README 写明"已验证 5/8 步"
- **方案 D**（极限兜底）：**6/12 决定降级**——只演示 CAW 单步 + 贡献提交 + 链上记账，跳过分账

**预防措施**：
- 6/10 强制做第一次端到端
- 每晚站会同步"今天串联到第几步"
- 留 1 个模块的"演示备份"

**Owner**：Hugo + 全体
**监控频率**：每日

## 三、🟡 黄色风险（次关键）

### R5 · Gas 估算偏差导致交易失败

**风险描述**：测试网 Gas 波动，transfer 失败

**应对**：
- 固定 gas price + 限速（避免拥堵时段）
- 加 retry 机制
- 失败时 fallback 用更高 gas

**Owner**：白织
**状态**：低概率，已规划应对

---

### R6 · x402 证明格式与 CAW Pact 衔接失败

**风险描述**：x402 标准细节、CAW Pact 字段映射可能有歧义

**应对**：
- 优先用 CAW 官方 Recipes（已列 30+）
- 大番薯 Day 1 调研 x402 协议规范
- 准备 "lite 模式"：不用真 x402，用简化 JSON 模拟

**Owner**：大番薯
**状态**：已规划调研

---

### R7 · 前端时间不够

**风险描述**：React Dashboard 是 P1，6/10 仍可能没雏形

**应对**：
- **降级方案 A**：录屏形式 + CLI 截图
- **降级方案 B**：用现成模板（Next.js + wagmi starter）
- **不做**：UI 美化（功能优先）

**Owner**：老实人
**状态**：降级方案已规划

---

### R8 · Telegram Bot API 限流

**风险描述**：Telegram Bot 推送过于频繁被限流

**应对**：
- 演示阶段只推送 1-2 次关键消息
- 用队列 + 退避策略
- 准备 Slack 备选通知渠道

**Owner**：老曹健身版
**状态**：低概率

---

### R9 · Demo Day 网络不稳定

**风险描述**：现场演示时 Sepolia/Monad RPC 抽风

**应对**：
- **全预录**（PRD 已明确）
- 准备本地 Hardhat node 离线备份
- 录屏时把 RPC URL 用 `localhost:8545` 演示

**Owner**：Hugo
**状态**：预案清晰

## 六、范围蔓延风险

### 6.1 容易蔓延的方向

- ❌ 做完整身份系统
- ❌ 做多链
- ❌ 做完整治理
- ❌ 做 Mobile App
- ❌ 做 KYC/合规

### 6.2 蔓延判断标准

每周复盘时问：
1. 这个功能对 Demo 有帮助吗？（无 → 砍）
2. 这个功能在 11 天内能完成吗？（否 → 砍）
3. 这个功能影响 4 大关键路径吗？（否 → 砍）

**砍单原则**：宁愿做 60% 完整 100%，不做 100% 完成 50%。

## 八、时间风险（11 天倒计时）

### 8.1 关键节点

```
6/6  第一堆火启动 → 如不启动 = 团队到齐失败
6/10 中间检查点   → 如端到端没跑通 = 启动降级方案
6/12 端到端完成   → 如未完成 = 录屏能跑部分
6/13 12:00 截止   → 死线
```

### 8.2 时间缓冲

| 阶段 | 计划天数 | 缓冲天数 | 实际可用 |
|------|----------|----------|----------|
| 设计 + 骨架 | 3 | 1 | 4 |
| 实现 + 测试 | 4 | 1 | 5 |
| 集成 + 优化 | 3 | 0.5 | 3.5 |
| 文档 + 录屏 | 2 | 0.5 | 2.5 |
| **合计** | **12** | **3** | **15** |

**实际有 4 天冗余**（11 天周期，15 天可用能力）→ 风险可控。
