# 创客星球 MVP 黑客松：贡献记录与价值分配 Agent 执行方案

> 会议同步版｜用于 2026-06-05 黑客松团队对齐  
> 版本：v0.1  
> 定位：先统一方向，再进入开发拆解

各位伙伴，今晚开会前，我先把这次创客星球 MVP 黑客松的核心方向和执行思路同步一下。

昨天晚上我们讨论了很多问题，但我感觉有一个核心还需要进一步对齐：

> 我们这次黑客松，不只是做一个技术 Demo，也是在真实实验创客星球未来的协作机制。

创客星球未来要解决的一个关键问题是：

> 在一个多人协作、边学边做、角色不断变化的团队里，怎么让每个人真实的贡献被看见、被记录、被证明，并最终用于价值分配？

所以，这次黑客松本身就是一次最小化实验。

## 0. 一图看懂核心闭环

```text

        ↓

        ↓

        ↓

```

一句话：

> **用 Agent 记录贡献，用 Wallet 分配价值。**

## 一、我们到底要做什么？

我们不是简单做一个“奖金分配表”。

我们要做的是一个：

> **面向 AI 协作团队的贡献记录与价值分配 Agent。**

更直白一点说：

> 把 Agent 拉进一个协作群里，它可以记录团队成员谁做了什么、谁响应了什么、谁补了什么位、谁交付了什么成果，然后根据提前设定的规则，生成贡献报告和分配建议。

这套机制先用于我们自己的黑客松团队。

如果这次能跑通，未来它可以外化给：

- 其他黑客松团队；
- DAO 组织；
- 开源项目；
- 创作者协作团队；
- 小型远程创业团队；
- AI 原生协作团队。

也就是说，我们不是只为自己分一次奖金，而是在做一个可以被其他团队使用的协作基础设施。

---

## 二、核心闭环

这次黑客松我们要验证的最小闭环是：

```text
团队协作过程
    ↓
Agent 自动/半自动记录贡献
    ↓
根据规则计算贡献分
    ↓
生成个人贡献证明和团队贡献报告
    ↓
输出奖金/收益/星钻分配建议
    ↓
通过 Cobo Agent Wallet 完成分配或模拟分配
```

一句话概括就是：

> **贡献 → 记录 → 证明 → 分配。**

这也是创客星球未来很重要的一条主线。

---

## 三、为什么这个方向重要？

传统团队协作里，贡献分配经常靠感觉。

容易出现几个问题：

1. 谁贡献大，说不清；
2. 谁临时补位，容易被忽略；
3. 谁拖延、无响应，也没有明确记录；
4. 只看代码量，不看沟通、协作、测试、文档、救火；
5. 到最后分奖金或权益时，容易产生误解。

但真实的团队贡献，不只是“谁写了多少代码”。

还有：

- 谁发现了关键问题；
- 谁推动了决策；
- 谁补上了别人没完成的部分；
- 谁让 Demo 最后能跑起来；
- 谁及时暴露风险，避免团队踩坑；
- 谁做了文档、测试、录屏、路演准备；
- 谁让整个团队更顺畅地协作。

这些都应该被记录。

---

## 四、这次贡献记录不是让大家手动填表

这里先说明一下：

这次不是要求每个人每天自己填复杂表格。

我的想法是：

> 大家正常在群里协作、认领任务、同步进度、提交结果。  
> 贡献记录由 Hermes / Agent 统一记录和整理。

也就是说，每个人不是“自己给自己记账”。

而是由 Agent 根据群聊、任务、文档、代码提交、测试结果等信息，统一做贡献记录。

大家只需要尽量做到：

1. 认领任务时，在群里说清楚；
2. 完成任务时，发出产出链接或截图；
3. 卡住时，及时说出来；
4. 帮别人补位时，也在群里留痕；
5. 重要结论尽量有记录。

这样 Agent 才能有依据。

## 五、贡献评估的五个参数

第一版我们不搞太复杂，先用五个维度来评估贡献。

### 1. 任务价值

看这个任务对 MVP 是否关键。

| 类型 | 说明 |
|---|---|
| 普通任务 | 辅助整理、一般资料、基础沟通 |
| 重要任务 | 前端页面、测试、文档、Demo 脚本 |
| 关键任务 | 合约、Agent、钱包、主流程打通 |
| 生死任务 | 没有它，项目无法提交或无法演示 |

### 2. 任务难度

看这个任务本身难不难。

包括：

- 技术难度；
- 协作难度；
- 不确定性；
- 是否需要跨模块配合；
- 是否容易卡住整体进度。

### 3. 完成度

不是只看“做了没做”，而是看完成到什么程度。

| 完成状态 | 说明 |
|---|---|
| 未开始 | 没有有效动作 |
| 有尝试 | 做了尝试，但没有可复用产出 |
| 部分完成 | 有部分结果，可以被别人接着用 |
| 基本完成 | 主体完成，还有小问题 |
| 完整完成 | 可以直接用于项目 |
| 超预期完成 | 不仅完成，还带来额外价值 |

### 4. 时间系数

时间节点很重要。

黑客松不是长期项目，很多事情晚一天就可能影响整体节奏。

所以同一个任务，不同完成时间，贡献权重应该不一样：

| 情况 | 处理原则 |
|---|---|
| 提前完成 | 加权 |
| 按时完成 | 正常计分 |
| 延迟但不影响整体 | 降低一部分权重 |
| 延迟并影响别人 | 明显降低权重 |
| 没完成但提前说明并交接 | 保留部分贡献 |
| 没完成且不说明 | 未完成部分不计分，必要时扣协作分 |

这里有一个重要原则：

> **提前暴露风险，比拖到最后更有价值。**

如果任务确实做不完，提前说出来，别人还能补位。  
这也是贡献。

### 5. 协作补位

这次我特别想把“协作贡献”记录进去。

因为黑客松能不能成，不只是技术问题，也是协作问题。

协作贡献包括：

- 主动同步进度；
- 主动暴露风险；
- 帮别人解决卡点；
- 临时接手没人做的任务；
- 在关键节点救火；
- 减少沟通内耗；
- 推动团队形成结论；
- 帮助 Demo 最终交付。

这些都应该算贡献。

## 六、第一版评分逻辑

第一版先用一个简单公式：

```text
任务贡献分 =
基础分 × 任务价值系数 × 难度系数 × 完成度系数 × 时间系数
```

然后再加上：

```text
最终贡献分 =
任务贡献分 + 协作加分 + 补位加分 - 阻塞扣分
```

这个公式不是为了机械算账，而是为了让贡献分配有一个相对透明的依据。

后面如果大家觉得某些权重不合理，可以继续调整。

## 七、任务没完成，怎么处理？

这个问题要提前说清楚。

### 情况一：没完成，但提前说明

如果一个任务没完成，但负责人提前说了：

- 当前进度到哪里；
- 卡点是什么；
- 哪部分可以交接；
- 需要谁补位；

那这不是简单的失败。

这种情况应该保留一部分贡献，因为他至少帮团队暴露了风险。

### 情况二：没完成，也不说明

如果任务到时间没完成，也没有提前同步，导致别人无法补位，影响整体进度。

那未完成部分不计分。  
如果明显影响别人，还应该扣协作分。

### 情况三：任务本身判断错了

如果不是个人不努力，而是技术路径本身走不通，那也不能简单归责给执行的人。

只要过程有记录、有反馈、有结论，这也算一种“风险验证贡献”。

因为他帮团队排除了错误路径。

### 情况四：别人临时补位完成

如果 A 没做完，B 临时补位完成。

那么：

- A 拿已完成部分的贡献；
- B 拿补位完成部分的主要贡献；
- 如果 B 是关键救火，还要额外加补位分。

这符合真实协作逻辑：

> 谁把项目往前推了，谁就应该被看见。

## 八、当前 MVP 要做到什么？

为了避免范围失控，这次黑客松第一版只做最小可行闭环。

### 我们要做到：

1. 记录团队协作中的贡献事件；
2. 按规则计算贡献分；
3. 生成每个人的贡献报告；
4. 输出团队贡献排名和分配建议；
5. 形成可解释的贡献证明；
6. 接入 Cobo Agent Wallet，完成一次分配或模拟分配；
7. 在 Demo 中讲清楚这个闭环。

### 暂时不做：

1. 不做复杂 SaaS 后台；
2. 不做多平台全自动接入；
3. 不做复杂申诉系统；
4. 不做所有 DAO 场景；
5. 不做过度精细的链上治理；
6. 不追求一开始就完全自动化。

这次先证明一件事：

> 一个团队可以让 Agent 参与协作过程，并根据真实贡献辅助价值分配。

## 九、产品未来可以怎么外化？

如果这次 MVP 跑通，未来它可以作为一个工具给其他团队使用。

大概流程是：

### 1. 项目方创建项目

填写：

- 项目名称；
- 成员列表；
- 奖金池/收益池；
- 协作群；
- 项目周期；
- 分配规则。

### 2. 选择贡献模板

| 模板 | 适用场景 |
|---|---|
| 黑客松模板 | 两周冲刺，快速交付 |
| DAO 治理模板 | 提案、投票、社区执行 |
| 开源项目模板 | issue、PR、review |
| 内容共创模板 | 文章、视频、设计、传播 |
| 商业项目模板 | 交付、运营、销售、技术 |

### 3. 拉 Agent 进协作群

Agent 开始记录：

- 谁认领任务；
- 谁完成任务；
- 谁提交代码；
- 谁写文档；
- 谁补位；
- 谁响应及时；
- 谁造成阻塞；
- 谁推动关键结果。

### 4. 生成贡献证明

最后每个人可以得到一份贡献报告：

```text
成员 A：
总贡献分：284
主要贡献：合约部署、钱包接入、Demo 修复
证据：GitHub commit、群聊记录、测试截图
建议分配比例：28.4%
```

### 5. 钱包执行分配

通过 Cobo Agent Wallet，根据贡献结果执行奖金、收益或积分分配。

这就是未来可以商业化的地方。

## 十一、今晚会议要确认什么？

今晚建议不要讨论太散，先确认五件事。

### 1. 是否认可这个产品方向？

也就是：

> 我们这次黑客松主线，是否聚焦在“贡献记录与价值分配 Agent”？

### 2. MVP 范围是否确认？

第一版是否只跑通：

```text
贡献记录 → 贡献评分 → 贡献证明 → 分配建议 → 钱包分配/模拟分配
```

### 3. 贡献参数是否先采用第一版？

也就是五个维度：

1. 任务价值；
2. 任务难度；
3. 完成度；
4. 时间系数；
5. 协作补位。

### 4. 开发分工是否围绕这个闭环调整？

我们不再分散做很多功能，而是围绕闭环拆：

- 合约 / 钱包；
- 前端展示；
- Agent 记录与评分；
- 文档 / Demo / 测试；
- 项目统筹与交付。

### 5. 今晚之后每个人要做什么？

会议结束后，最好每个人都能明确：

- 我负责哪一块；
- 今天/明天交付什么；
- 什么时间同步；
- 卡住找谁；
- 什么算完成。

## 十二、我对大家的期待

这次黑客松，我们都是边学边做，不可能一开始什么都完美。

所以我更看重三件事：

1. **及时同步**
   - 做了什么，说出来；
   - 卡在哪里，说出来；
   - 需要谁帮忙，说出来。

2. **主动补位**
   - 看到没人做的事，可以主动接；
   - 看到别人卡住，可以帮一把；
   - 看到方向不清楚，可以提出来讨论。

3. **结果意识**
   - 不追求一开始做大而全；
   - 先把最小闭环跑通；
   - 先保证最后能 Demo、能提交、能讲清楚。

这次不是谁给谁打工，也不是谁指挥谁。

我们是在一起实验一种新的协作方式。

如果这个闭环跑通，它就是创客星球未来很重要的基础能力。
